# ActionCap 浏览器插件设计方案

## 1. 目标概述

ActionCap 是一个面向 Chrome / Edge 的操作录制插件，用于记录用户在浏览器页面中的操作轨迹，以及页面内产生的请求与响应内容，并在录制结束后生成可检索、可分析、可回放的结果页面。

核心目标：

1. 用户点击浏览器右上角插件 icon，弹出控制面板。
2. 用户可选择录制范围：仅当前 Tab、跨 Tab、所有窗口。
3. 用户可开始录制、结束录制。
4. 结束后自动打开结果页面。
5. 结果页面可按时间顺序展示各 Tab 中的用户操作。
6. 结果页面可按时间顺序展示各 Tab 中的请求与响应内容。
7. 用户点击“回放”后，可进入可视化回放界面查看整个操作过程。

本方案以 **Chrome Extension Manifest V3** 为基础，兼容 **Chrome / Edge（Chromium 内核）**。

---

## 2. 产品定位

### 2.1 适用场景

1. 复现线上问题时记录用户真实操作链路。
2. 采集接口请求与响应，辅助排查前后端交互问题。
3. 记录多 Tab 工作流，分析业务操作步骤。
4. 将一次用户操作过程沉淀为可复查的“浏览器会话”。

### 2.2 产品原则

1. **显式录制**：只有用户手动点击开始后才录制。
2. **按范围录制**：用户明确选择录制边界，避免无关页面进入记录。
3. **时间线统一**：操作、请求、响应全部统一到同一条时间轴。
4. **结果先可读，再可回放**：先满足排查问题的可读性，再满足直观回放。
5. **隐私可控**：敏感输入、Cookie、Authorization 等默认脱敏。

---

## 3. 信息架构

插件只有两个一级界面：

1. **控制弹窗 Popup**
2. **结果页面 Results**

其中“回放界面”不单独作为第三级产品，而是作为 **结果页面中的一个视图模式**，由“回放”按钮进入。

建议路由：

1. `popup.html`
2. `results.html?sessionId=xxx`
3. `results.html?sessionId=xxx&view=replay`

---

## 4. 交互流程

### 4.1 开始录制流程

1. 用户点击插件 icon。
2. 弹出控制面板。
3. 用户选择录制范围：
   - 仅当前 Tab
   - 跨 Tab
   - 所有窗口
4. 用户点击“开始录制”。
5. 插件创建一个新的录制会话 `session`。
6. 后台服务开始：
   - 注入内容脚本，监听用户页面行为
   - 连接目标 Tab 的调试通道，采集请求与响应
   - 持续将数据写入 IndexedDB
7. Popup 进入“录制中”状态。

### 4.2 结束录制流程

1. 用户再次点击插件 icon，打开 Popup。
2. 用户点击“结束录制”。
3. 插件停止所有监听、解绑调试连接、封存会话。
4. 插件自动新开一个标签页或窗口打开结果页。
5. 结果页默认展示“时间线视图”。

### 4.3 回放流程

1. 用户在结果页点击“回放”。
2. 页面切换至“回放视图”。
3. 回放区按时间推进渲染用户页面操作。
4. 若跨 Tab 录制，则时间轴走到其他 Tab 的事件时自动切换目标 Tab 视图。

---

## 5. 录制范围定义

为避免歧义，建议将三种范围明确定义如下：

### 5.1 仅当前 Tab

只录制开始时所在的那个 Tab 及其后续导航。

包含：

1. 当前 Tab 中的点击、输入、滚动、跳转
2. 当前 Tab 中发出的请求与响应
3. 当前 Tab 内 iframe 的事件和请求

不包含：

1. 用户切换到其他 Tab 后的操作
2. 其他窗口中的行为

### 5.2 跨 Tab

录制当前窗口中的多个 Tab。

包含：

1. 当前窗口中用户切换到的 Tab
2. 当前窗口中新打开的 Tab
3. 这些 Tab 内的操作与请求/响应

不包含：

1. 其他浏览器窗口中的 Tab

### 5.3 所有窗口

录制所有普通浏览器窗口中的所有可跟踪 Tab。

包含：

1. 所有窗口中的切换、打开、关闭
2. 所有被纳入范围的 Tab 内操作与请求/响应

不包含：

1. `chrome://`、`edge://`、扩展页、浏览器内部页面
2. 浏览器不允许注入脚本或附加调试器的特殊页面

## 6. 界面设计

### 6.1 总体视觉方向

建议采用 **“取证控制台 / Forensic Console”** 风格，而不是普通开发者工具式白底表格。

视觉关键词：

1. 深石墨底色
2. 荧光绿与琥珀色状态强调
3. 宽大数字和等宽文本混排
4. 具有“录制会话”气质的时间线与状态指示

建议配色：

1. 背景：`#0D1117`
2. 面板：`#151B23`
3. 主强调：`#4CFF93`
4. 次强调：`#FFB84D`
5. 危险：`#FF5D73`
6. 文本主色：`#E6EDF3`
7. 文本次色：`#8B949E`

建议字体：

1. 标题：`Sora` 或 `IBM Plex Sans`
2. 正文：`IBM Plex Sans`
3. 数据与代码：`IBM Plex Mono`

### 6.2 Popup 控制弹窗

### 目标

让用户在 5 秒内完成：

1. 识别当前状态
2. 选择录制范围
3. 开始或结束录制

### 布局

建议尺寸：`360 x 520`

结构分为 4 个区块：

1. 顶部状态区
2. 录制范围区
3. 主操作区
4. 风险提示区

### 区块设计

#### 1. 顶部状态区

内容：

1. 插件名 `ActionCap`
2. 当前状态标签
   - `Idle`
   - `Recording`
   - `Stopping`
3. 当前会话摘要
   - 已录制时长
   - 已纳入 Tab 数
   - 已捕获事件数

#### 2. 录制范围区

使用卡片式三选一，而不是普通单选框。

三个选项：

1. `当前 Tab`
   - 描述：只记录当前标签页及其导航
2. `跨 Tab`
   - 描述：记录当前窗口中的多个标签页
3. `所有窗口`
   - 描述：记录所有窗口中的全部标签页

视觉上用卡片高亮当前选择，右上角显示范围图标。

#### 3. 主操作区

按钮：

1. 主按钮 `开始录制`
2. 录制中时切换为 `结束录制`

辅助信息：

1. 录制开始时间
2. 当前目标窗口 / Tab 标识
3. 是否已成功连接网络采集器

#### 4. 风险提示区

说明：

1. 将记录页面操作与网络内容
2. 敏感字段默认脱敏
3. 某些页面因浏览器限制无法采集

### Popup 状态变化

#### Idle

1. 范围选择可编辑
2. 主按钮为绿色

#### Recording

1. 范围选择锁定
2. 主按钮改为红色 `结束录制`
3. 显示跳动录制指示点

#### Stopping

1. 按钮禁用
2. 文案：`正在整理结果...`

### 6.3 结果页面 Results

### 页面目标

让用户能快速回答 3 个问题：

1. 用户做了什么？
2. 页面发了什么请求，收到了什么响应？
3. 这些事件发生时页面状态是什么？

### 页面布局

建议采用三栏布局：

1. 左栏：会话与 Tab 目录
2. 中栏：时间线列表
3. 右栏：详情面板

顶部放全局工具条。

### 顶部工具条

内容：

1. 会话名称
2. 录制范围标签
3. 开始时间 / 总时长
4. 搜索框
5. 过滤器
   - 全部
   - 仅操作
   - 仅请求
   - 仅响应
   - 错误
6. `回放` 按钮
7. `导出 JSON` 按钮
8. `删除会话` 按钮

### 左栏：会话与 Tab 目录

展示内容：

1. 会话元信息
2. 录制统计
   - Tab 数
   - 操作数
   - 请求数
   - 错误数
3. Tab 列表
   - favicon
   - 标题
   - URL
   - 每个 Tab 的事件数量

点击某个 Tab 后，中栏切换为该 Tab 的时间线；也可保留“全部 Tab”视图。

### 中栏：时间线列表

结果页最核心的区域。

按时间顺序统一渲染三种事件：

1. 用户操作事件
2. 网络请求事件
3. 网络响应事件

每条记录需要有：

1. 时间戳
2. Tab 标识
3. 事件类型图标
4. 一行摘要
5. 可展开详情

#### 用户操作事件示例

1. 点击了按钮：`button.submit-order`
2. 在输入框中输入：`邮箱`
3. 滚动到：`Y = 2480`
4. 页面导航到：`/checkout`
5. 切换到 Tab：`订单确认`

#### 网络事件示例

1. `POST /api/order/create`
2. `200 OK /api/order/create`
3. `GET /api/profile`
4. `500 Internal Server Error /api/profile`

### 右栏：详情面板

当用户点击中栏的一条时间线记录时，右栏展示详情。

#### 若为操作事件

展示：

1. 操作类型
2. 页面 URL
3. 目标元素信息
   - 标签名
   - 文本摘要
   - selector
   - 坐标
4. 输入值
   - 默认脱敏
5. 对应截图时间点或回放跳转按钮

#### 若为请求事件

展示：

1. 方法
2. URL
3. 发起时间
4. 请求头
5. Query 参数
6. Request Body
7. 发起来源
   - document / fetch / xhr / script / image

#### 若为响应事件

展示：

1. 状态码
2. 耗时
3. Response Headers
4. Response Body
5. MIME Type
6. 是否截断
7. 是否 base64

---

### 6.4 回放视图 Replay

回放视图作为结果页的第二视图。

### 布局

建议采用：

1. 顶部：播放控制条
2. 中间：页面回放舞台
3. 底部：时间轴
4. 右侧：当前事件详情

### 播放控制条

包含：

1. 播放 / 暂停
2. 上一步 / 下一步
3. 播放速度 `1x / 2x / 4x`
4. 跳转到指定时间
5. 当前 Tab 标签
6. 跟随全局时间开关

### 页面回放舞台

建议使用 **DOM 快照回放**，而不是简单截图翻页。

能力要求：

1. 回放用户点击、输入、滚动、页面变化
2. 显示鼠标指针运动与点击脉冲
3. 显示当前 URL
4. 时间点切换到其他 Tab 时自动切换舞台

### 底部时间轴

时间轴上要标记不同事件类型：

1. 操作用绿色点
2. 请求用蓝色点
3. 响应用琥珀色点
4. 错误用红色块

点击任意标记，舞台跳转到对应时刻。

### 右侧事件详情

展示当前播放时刻对应的：

1. 操作说明
2. 关联请求
3. 关联响应
4. 当前 Tab 信息

## 7. 技术方案

### 7.1 技术选型

推荐：

1. **Manifest V3**
2. **TypeScript**
3. **React + Vite** 构建 Popup 与 Results 页面
4. **IndexedDB + Dexie** 存储录制会话
5. **rrweb** 录制与回放 DOM 变化
6. **chrome.debugger API** 捕获请求与响应体

原因：

1. Popup / Results 页面开发效率高
2. `rrweb` 适合做可视化操作回放
3. `chrome.debugger` 是扩展中拿到请求体和响应体的可行方案之一

### 7.2 插件架构

建议模块结构：

```text
src/
  background/
    service-worker.ts
    session-manager.ts
    debugger-recorder.ts
    tab-scope-manager.ts
  content/
    recorder.ts
    action-capture.ts
    rrweb-adapter.ts
  injected/
    page-bridge.ts
  popup/
    popup.html
    popup.tsx
  results/
    results.html
    results.tsx
    replay.tsx
  common/
    types.ts
    events.ts
    selectors.ts
    sanitizer.ts
    storage.ts
    time.ts
manifest.json
```

### 各模块职责

#### 1. Background Service Worker

负责：

1. 创建和结束录制会话
2. 维护当前录制状态
3. 管理目标 Tab 范围
4. 为目标 Tab 附加 / 解除 `chrome.debugger`
5. 接收内容脚本发来的操作事件
6. 统一写入存储
7. 停止后打开结果页

#### 2. Content Script

负责：

1. 监听用户行为
2. 收集元素信息
3. 启动 `rrweb` 录制 DOM 变化
4. 将事件持续发送给 Background

#### 3. Injected Page Bridge

用于在页面主世界中补充采集能力。

可选职责：

1. 捕获 SPA 路由变化
2. 监听页面级自定义事件
3. 在必要时补充 `fetch/XHR` 语义信息

#### 4. Popup

负责录制控制，不负责展示大结果。

#### 5. Results

负责：

1. 加载并解析会话
2. 渲染时间线
3. 提供检索 / 过滤 / 导出
4. 切换到回放视图

## 8. 数据采集设计

### 8.1 用户操作采集

### 建议记录的事件

1. `click`
2. `dblclick`
3. `contextmenu`
4. `input`
5. `change`
6. `submit`
7. `keydown` / `keyup`
8. `focus` / `blur`
9. `scroll`
10. `selectionchange`
11. `navigation`
12. `tab-activated`
13. `tab-created`
14. `tab-closed`
15. `history pushState / replaceState / popstate`

### 每个操作事件字段

```ts
type UserActionEvent = {
  id: string
  sessionId: string
  tabId: number
  frameId: number
  ts: number
  type: 'click' | 'input' | 'scroll' | 'navigation' | 'tab-activated' | ...
  url: string
  title?: string
  selector?: string
  element?: {
    tagName?: string
    text?: string
    id?: string
    className?: string
    role?: string
    name?: string
  }
  coordinates?: { x: number; y: number }
  scroll?: { x: number; y: number }
  value?: string
  masked?: boolean
  rrwebEventId?: string
}
```

### 元素定位策略

为了让结果页可读且回放有指向性，建议保留“稳定 selector”：

优先级：

1. `data-testid`
2. `id`
3. `name`
4. `aria-label`
5. 文本摘要 + 结构路径
6. CSS path 兜底

同时保留：

1. 元素文本摘要
2. 坐标
3. 元素矩形位置

这样即使 selector 失效，结果列表仍然可读。

### 8.2 网络请求与响应采集

这是整个方案中最关键、最容易踩坑的部分。

### 推荐方案

使用 `chrome.debugger` 连接目标 Tab，并接入 Chrome DevTools Protocol 的 `Network` 域。

采集链路：

1. `Network.enable`
2. 监听 `Network.requestWillBeSent`
3. 监听 `Network.responseReceived`
4. 监听 `Network.loadingFinished`
5. 在适当时机调用：
   - `Network.getRequestPostData`
   - `Network.getResponseBody`

### 为什么不用单纯 `webRequest`

`webRequest` 适合拿到请求生命周期和头部，但 **无法稳定获取响应体内容**。而用户需求明确要求记录“请求和响应内容”，所以核心采集链路应基于 `chrome.debugger`。

### 网络事件字段

```ts
type NetworkEvent = {
  id: string
  sessionId: string
  tabId: number
  frameId?: string
  requestId: string
  ts: number
  phase: 'request' | 'response'
  url: string
  method?: string
  status?: number
  statusText?: string
  resourceType?: string
  mimeType?: string
  requestHeaders?: Record<string, string>
  responseHeaders?: Record<string, string>
  requestBody?: string
  responseBody?: string
  bodyEncoding?: 'plain' | 'base64'
  durationMs?: number
  initiator?: string
  truncated?: boolean
  errorText?: string
}
```

### 重要限制

1. 并不是所有响应体都适合保存，二进制资源应默认跳过或仅保留摘要。
2. 超大响应体应截断，例如默认只保留前 `1 MB`。
3. 流式响应、下载流、媒体流不保证完整。
4. 某些跨进程页面或浏览器内部页面无法附加调试器。
5. 使用 `debugger` 权限时，用户会感知到浏览器的调试提示，这是需要接受的产品代价。

### 建议的采集规则

默认记录：

1. `document`
2. `fetch`
3. `xhr`
4. `script`
5. `stylesheet`

可配置记录：

1. `image`
2. `font`
3. `media`

默认对以下内容做保护：

1. `Authorization`
2. `Cookie`
3. `Set-Cookie`
4. 密码字段
5. 支付、身份证、手机号等敏感信息

---

### 8.3 DOM 回放采集

为了实现可视化回放，建议使用 `rrweb`。

录制内容：

1. 初始全量 DOM 快照
2. DOM 增量变更
3. 鼠标移动
4. 点击
5. 输入
6. 滚动
7. 视口变化

### 为什么选择 rrweb

1. 已有成熟的录制与回放能力
2. 适合在结果页直接重建页面过程
3. 可以将“回放界面”与“事件列表”在同一时间轴上对齐

### 与业务事件的关系

ActionCap 需要同时保存两类数据：

1. `rrweb` 原始事件，用于回放
2. 语义化操作事件，用于结果列表与检索

二者通过时间戳关联，不建议互相替代。

## 9. 存储设计

### 9.1 存储方式

使用 `IndexedDB`，不建议把完整数据塞进 `chrome.storage.local`。

原因：

1. 会话数据量可能很大
2. DOM 快照和响应体都可能占用较多空间
3. IndexedDB 更适合结构化查询和分表存储

推荐表结构：

1. `sessions`
2. `tabs`
3. `user_actions`
4. `network_events`
5. `replay_events`

---

## 10. 统一事件时间线

结果页不能把“操作列表”和“网络列表”割裂成两个孤立世界，建议统一成一个基础时间轴，再派生多个视图。

统一事件结构：

```ts
type TimelineEvent =
  | { kind: 'user-action'; ts: number; tabId: number; refId: string }
  | { kind: 'network-request'; ts: number; tabId: number; refId: string }
  | { kind: 'network-response'; ts: number; tabId: number; refId: string }
  | { kind: 'system'; ts: number; tabId?: number; refId: string }
```

在 UI 上可以有两个列表视图：

1. **统一时间线视图**
2. **分组视图**
   - 按 Tab 分组
   - 按操作 / 请求 / 响应分组

但底层数据应该共享同一时间坐标系。

## 11. 跨 Tab / 跨窗口的会话管理

### 11.1 Session Manager

后台需要有一个 `SessionManager` 负责：

1. 会话创建
2. 范围判断
3. Tab 纳入与剔除
4. debugger 附加状态维护
5. 会话结束与汇总

### 11.2 范围判断逻辑

### 当前 Tab 模式

只追踪 `startTabId`。

### 跨 Tab 模式

追踪 `startWindowId` 下用户激活、打开、跳转到的 Tab。

### 所有窗口模式

追踪所有普通浏览器窗口内的可采集 Tab。

### 11.3 Tab 生命周期事件

建议监听：

1. `tabs.onActivated`
2. `tabs.onCreated`
3. `tabs.onRemoved`
4. `tabs.onUpdated`
5. `windows.onFocusChanged`
6. `webNavigation.onCommitted`

这些事件也应进入系统时间线，便于结果页解释“为什么切换到了另一个 Tab”。

## 12. 权限设计

建议 `manifest.json` 权限：

```json
{
  "manifest_version": 3,
  "name": "ActionCap",
  "permissions": [
    "storage",
    "tabs",
    "scripting",
    "debugger",
    "webNavigation"
  ],
  "host_permissions": ["<all_urls>"]
}
```

可选权限：

1. `downloads`
   - 用于导出 JSON
2. `activeTab`
   - 作为最小授权辅助

### 权限说明

1. `debugger`：抓取请求体与响应体的关键权限
2. `scripting`：动态注入内容脚本
3. `tabs`：管理 Tab 与打开结果页
4. `webNavigation`：跟踪导航与页面切换
5. `<all_urls>`：只有这样才能覆盖多站点录制

## 13. 隐私与安全

这是产品上线前必须明确的一部分。

### 默认脱敏策略

1. 输入框中 `type=password` 一律不记录原文
2. 命中关键词的字段脱敏：
   - password
   - token
   - authorization
   - cookie
   - secret
   - phone
   - idcard
3. 请求头中的敏感项默认替换为 `***`
4. 响应体中若识别为二进制或超大内容，默认截断

### 用户感知

Popup 中明确提示：

1. 正在记录页面操作和网络内容
2. 某些敏感信息已自动脱敏
3. 浏览器内部页面不支持采集

### 数据留存

建议：

1. 默认只保留本地
2. 不自动上传任何远端服务
3. 提供“删除会话”能力
4. 后续可增加“自动清理 7 天前会话”

---

## 14. 性能与稳定性策略

### 14.1 性能控制

1. 高频事件做采样或节流
2. 响应体设置大小上限
3. 图片、字体默认不保存 body
4. 数据到达即写入 IndexedDB，避免只放内存

### 14.2 MV3 Service Worker 休眠问题

Manifest V3 的后台是 Service Worker，不能假设它一直常驻。

所以设计上应做到：

1. 录制状态持久化到 IndexedDB 或 `chrome.storage.session`
2. debugger 附加关系可恢复
3. 收到事件时可重新装载上下文

不要把“当前录制会话”只放在内存变量里。

### 14.3 降级策略

若某个 Tab 无法附加 debugger：

1. 仍可继续记录用户操作
2. 网络区显示“该页面未成功接入网络采集”

若某个页面无法注入内容脚本：

1. 仍记录 Tab 激活、导航等系统事件
2. 回放区显示“该页面不支持可视化回放”

## 15. 结果页的核心视图设计建议

为了提高实用性，结果页不要只堆技术字段，建议内建 3 种视图：

### 15.1 全部时间线

适合快速看全局流程。

### 15.2 Tab 视图

按 Tab 切换，适合排查跨页、多标签流程。

### 15.3 请求诊断视图

按请求聚合，展示：

1. 请求摘要
2. 响应摘要
3. 前后关联操作

例如：

`点击“提交订单” -> POST /api/order/create -> 500 错误`

这种视图对问题排查价值很高。

## 18. 关键风险

### 风险 1：`debugger` 权限带来的用户感知

这是换取响应体采集能力的成本，产品文案必须提前说明。

### 风险 2：数据量过大

DOM 快照和响应体都可能很重，必须做截断、压缩、类型过滤。

### 风险 3：特殊页面无法采集

浏览器内部页、扩展页、部分受限站点可能无法完整记录，需要在 UI 中明确提示，而不是静默失败。

### 风险 4：跨 Tab 回放复杂度高

建议第一版回放中支持“按全局时间自动切换 Tab”，而不是追求多舞台同时回放。

## 19. 最终方案总结

ActionCap 的推荐实现方案是：

1. 使用 **MV3** 构建插件基础能力。
2. 使用 **Popup** 作为录制控制入口。
3. 使用 **Results 页面** 作为结果分析与回放入口。
4. 使用 **content script + rrweb** 记录用户可视化操作。
5. 使用 **chrome.debugger + CDP Network** 捕获请求与响应内容。
6. 使用 **IndexedDB** 持久化会话、操作、网络、回放事件。
7. 使用 **统一时间线** 将多 Tab、多窗口、多类型事件组合成一个可分析的浏览器会话。

如果按这个方案实施，插件不仅能满足“记录用户操作 + 请求/响应内容 + 回放”的需求，而且技术路径是现实可行的，不会停留在概念稿层面。
